# Agent.md

## 目的

本文件用于沉淀当前项目中与用户协作相关的稳定工作规则，作为后续 Codex / Agent 执行任务时必须遵守的约束。

它补充并提炼自 `CLAUDE.md` 中的“用户习惯”与已确认协作流程，优先级高于临时聊天中的含糊推断。

## 用户工作流

用户的标准流程如下：

1. 用户先向 Claude 提需求。
2. Claude 先提供方案，而不是直接进入编码。
3. 方案经过多轮沟通和确认后定稿。
4. 定稿后产出一份项目级设计方案，作为后续开发的基线。
5. Codex 基于这份已确认的设计方案开始开发。
6. 如果开发过程中出现关键修改点，必须先同步更新设计方案，再继续实现。

## 设计与实现规则

- 设计方案是开发基线，不允许绕开设计文档自行改主方向。
- 关键方案变化必须回写设计文档，不能只留在聊天记录里。
- 以下内容默认属于“关键方案变化”：
- 页面结构或核心交互流程变化
- 后端架构变化
- API 契约变化
- 数据模型变化
- 多智能体分工方式的根本变化
- 任何会影响最终产品边界、体验预期或系统行为的调整

## 多 Agent 策略

- Codex 可以自行决定是否创建多 agent。
- 是否创建多 agent 的判断标准不是“形式上更复杂”，而是“是否能明显提效、提质量”。
- 只有在任务边界清晰、并行不会引入额外混乱、且结果能可靠整合时，才应主动使用多 agent。
- 如果多 agent 会增加协调成本、降低一致性或影响设计落地，应优先单线程高质量推进。

## 开发与交付要求

- Codex 开发完成后，必须先做全面自测，再交付给用户。
- 不允许只完成代码修改就宣称“已经好了”。
- 自测应覆盖至少以下内容：
- 构建或编译
- 关键主流程验证
- 与本次改动直接相关的回归检查
- 对外说明时必须明确区分：
- 哪些已经真实验证
- 哪些只是推断
- 哪些仍有风险或待验证

## 沟通规则

- 如果存在不确定点，允许并鼓励与用户充分沟通，不要闷头假设。
- 需要确认时，应聚焦在真正影响方案或结果的关键问题上。
- 一旦用户已经明确表达设计意图，后续实现应尽量贴近该意图，不要自行“优化”成另一套方案。

## 严格禁止事项

- 不允许为了省事随意加入兜底或降级策略，导致结果偏离用户设计初衷。
- 不允许用写死、伪计划、伪 agentic、假数据或伪执行来冒充真实能力。
- 不允许把“暂时可跑”当成“符合设计”。
- 不允许把关键决策只记录在聊天里而不更新设计文档。

## 当前项目中的补充偏好

- 用户非常重视“项目方案和进展持续沉淀”。
- 新对话开始时，应主动恢复项目方案、当前状态和下一步建议。
- 每次有实质进展后，应及时更新交接文档。
- 记录时要明确区分“本轮已验证”和“仅计划”。
- 优先展示计划全景，再执行。
- 执行动作要有清晰中文语义说明。
- 尽量避免日志流式、不断下滚的体验，优先使用结构化面板、计划图和原地更新。

## 执行口径

后续若 `agent.md`、设计文档、交接文档之间出现冲突，应按以下顺序处理：

1. 已经被用户明确确认的最新设计方案
2. `agent.md` 中的协作规则
3. 本轮真实验证得到的运行事实
4. 历史交接文档中的旧计划或旧判断
